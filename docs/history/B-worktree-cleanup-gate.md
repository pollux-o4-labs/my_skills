# B-worktree-cleanup-gate — 설계 근거

2026-07-15. `AIL-worktree-parallel-guard/hooks/` 훅의 grounds. 훅 주석·SKILL.md 는 현재 상태만 말하고, 왜 그 형태인지는 여기를 참조한다.

## 문제: 규칙이 아니라 소환 시점이 없었다

vgo 에서 머지된 워크트리가 개당 5.2G 씩 쌓였다(조사 중 4개 → 9개, 약 47GB). 규칙은 이미 두 곳에 있었다 — vgo 규칙 06 제2조("감독은 워크트리 뮤테이션을 전담해야 한다")와 인계 문서("개당 5.2G, 끝나면 반드시 정리"). 글이 없어서 안 지킨 게 아니다. 둘 다 "작업 끝나면"이라고만 해서, `worktree add` 때는 규칙이 소환되고 `remove` 때는 아무 일도 안 일어났다. 처방은 문장 추가가 아니라 트리거였다.

## 폐기 1: 로컬 머지 판정 — 실측 진짜양성 0%

초안은 `git merge-base --is-ancestor <branch> HEAD` 로 "머지된 워크트리"를 골랐다. vgo 에서 실측한 결과 **맞힌 게 하나도 없었다**.

- `gh pr merge` 는 서버에서 머지한다. 로컬 HEAD 는 그 사실을 모른다 — 방금 머지한 브랜치를 원리적으로 못 잡는다.
- squash·rebase 머지는 조상 관계를 아예 남기지 않는다. 로컬 판정이 성립할 수 없다.
- 그러면서 오탐 2종을 냈다. **주 체크아웃(5.5G)을 삭제 권고** — `git rev-parse --show-toplevel` 은 워크트리 상대적이라 워크트리 안에서 실행하면 자기 자신을 가리키고, "주 체크아웃 제외" 가드가 무력화된다. **갓 만든 워크트리를 머지됨으로 오탐** — `--is-ancestor` 는 브랜치가 HEAD 와 같은 해시여도 참이다. 병렬 워크트리를 지원하려고 존재하는 스킬이 그 워크트리를 지우라고 하는 꼴이었다.

"놓치는 쪽이 오탐보다 낫다"고 주석에 정당화해뒀는데, 실제로는 놓치면서 동시에 오탐했다.

## 폐기 2: 서버 조회 — 정확하지만 과구현

`gh pr view <n> --json headRefName,mergedAt,state` 는 **0.45초에 브랜치명을 정확히 준다**(vgo PR #9 → `docs/issue-template` = `vgo-wt-form`). squash 가 파괴하는 건 로컬 조상이지 서버의 `mergedAt` 이 아니다. 판정 불가는 로컬의 한계일 뿐 사실이 아니었다.

그럼에도 뺐다. 조회를 넣으면 워크트리 열거(`-z`·detached·prunable)·크기 측정·삭제 명령 조립·`-D` 가드가 줄줄이 따라붙어 225줄이 됐다. **어느 워크트리가 끝났는지는 방금 머지한 감독이 이미 안다.** 판정하지 않으면 오판할 것도 없고, 소환 한 줄이면 되는 일이었다. 84줄로 축소.

부수 효과: `du -sh` 제거로 5.1s(cold) → 36ms.

## 확정된 계약

- **`exit 0` + `hookSpecificOutput.additionalContext`.** 공식 문서상 PostToolUse 는 차단 불가이고 `exit 2` 는 stderr 를 *에러 메시지로* 전달한다 — 성공한 머지를 실패로 오해시키고 stdout JSON 도 통째로 무시된다. 두 경로는 상호배타적이다.
- **판정·삭제 없음.** 워크트리 뮤테이션은 감독 전담(vgo 규칙 06 제2조). 훅이 remove 를 직접 치면 그 규칙을 훅이 어기는 자기모순이고, dirty 워크트리나 주 체크아웃을 날릴 위험을 떠안는다.
- **브랜치 삭제는 안내조차 안 한다.** 회수 대상인 GB 는 워크트리에 있고 브랜치 ref 는 0바이트다. `-d` 는 squash 머지에서 반드시 실패하고 git 이 친절히 `-D` 를 안내해 근거 없는 강제삭제로 승격시킨다. 서버가 MERGED 라 해도 미푸시 로컬 커밋이 있으면 `-D` 가 그것을 조용히 파괴한다(실측). 얻을 것 없이 잃을 것만 있다.
- **cwd 워크트리 경고 한 줄은 남긴다.** clean 한 cwd 워크트리는 `git worktree remove` 가 거부 없이 성공(rc=0)해 디렉토리가 증발하고, 이후 모든 명령이 `Unable to read current working directory` 로 깨진다. git 이 막아주지 않는다.

## 파서: 렉서에 구분자 분리까지 맡긴다

정규식으로 먼저 쪼개고 나중에 shlex 를 돌리면 인용문 안의 `;`·개행이 분리를 만든다 — `git commit -m "fix; gh pr merge; done"` 이 오탐 발화한다(초안의 실제 결함이자, 자매 훅 `git-workflow-gate.sh` 가 지금도 가진 구멍). `shlex.shlex(posix=True, punctuation_chars=";|&\n")` + `whitespace=" \t\r"` 로 렉서가 구분자까지 토큰화하게 하면 인용문이 보호된다. heredoc 본문은 명령이 아니라 데이터라 사전 제거한다.

회귀 19케이스 통과 — 오탐(heredoc 4종·인용문·`--body` 개행·주석·서브셸·`pr create`) 침묵, 진짜양성(`-R` 선행형·개행체인·절대경로·번호생략·`--auto`) 발화.

## 인터프리터: python3 우선 (자매 훅 무음 고장의 근인)

데비안·우분투 계열은 `python3` 만 깔고 `python` 이름은 `python-is-python3` 를 따로 설치해야 생긴다. `command -v python` 으로만 찾으면 훅이 판정을 포기하고 **상시 무음 통과**한다.

이건 가정이 아니다. 자매 훅 `git-workflow-select/hooks/git-workflow-gate.sh` 가 정확히 이 함정에 빠져 있다 — 마커 없는 repo 에서 `git switch -c dev` → **exit 0**(차단돼야 함) 실측. 스스로 "스킬 문서로는 못 막는 유일한 하드 게이트"라 적어둔 훅이 아무것도 막지 못하고 있었고, `~/.claude/settings.json` 에 등록조차 안 돼 있었다. **미수정 — 별건.**

교훈: 안전판("판정 불가면 통과")이 곧 무음 실패 경로다. 조건이 상시 참이면 훅은 있으나 마나다.
