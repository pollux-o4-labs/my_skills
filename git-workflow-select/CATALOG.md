# CATALOG — 워크플로 선택지 (사용자 제시용)

마커 없는 repo 에서 사용자에게 워크플로를 물을 때, **이 파일을 그대로 건넨다.** id·구조·적합 상황을 한눈에.

```
┌─────────────────────────────────────────────────────────────┐
│ 1) github-flow          [1인·소규모 기본 추천]               │
│                                                              │
│   main ──●──────●──────●────►                                │
│           \    /  \    /                                     │
│         feature  feature      (짧게 파고 PR→squash 머지)     │
│                                                              │
│   → 지속 배포, 릴리스 = main 태그. 오버헤드 최소.            │
└─────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────┐
│ 2) git-flow             [릴리스 버전 관리 필요]              │
│                                                              │
│   main ────●─────────────●────►   (배포본 + 태그)            │
│            │             │                                   │
│   release  │        ┌──release/1.0                           │
│            │        │                                        │
│   develop ─●──●──●──●──●────►      (통합)                    │
│              \  /                                            │
│            feature                                           │
│   hotfix ──(긴급 시 main에서 직접)                           │
│                                                              │
│   → 배포 주기 명확·버전 태깅 필수일 때. 1인이면 과할 수도.   │
└─────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────┐
│ 3) trunk-based          [빠른 통합·CI 강함]                  │
│                                                              │
│   main ─●●●●●●●●●────►   (하루 안에 통합, 미완성은 flag 뒤)  │
│          ││││                                                │
│         짧은 작업 브랜치 (수시간)                            │
│                                                              │
│   → CI 강하고 리뷰 가벼울 때. 브랜치 오래 안 끈다.           │
└─────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────┐
│ 4) middle-merge         [서브에이전트 병렬·화면검증]         │
│                                                              │
│   main ─────────────────●────►     (squash 1줄만)            │
│                         │                                    │
│   middle-merge ─●───●───●────►     (영구, 통합 검증 게이트)  │
│                 │   │                                        │
│   integration/  ●   ●              (타입별 bucket)           │
│      feat      /│                                            │
│   feat/topic ─● │  ← worktree 격리                          │
│   fix/sub ─────●   ← worktree 격리 (시행착오)               │
│                                                              │
│   → 1인이 여러 에이전트 결과물을 실측 검증, main history     │
│     청결 필요할 때. 상세 규칙 = efficient-subagent 스킬.     │
└─────────────────────────────────────────────────────────────┘
```

**고르는 법**: 위→아래로 갈수록 단순(1인용)→복잡(병렬·대규모). 애매하면 1인 기획/코드는 `github-flow`, 서브에이전트 병렬 대작업은 `middle-merge`.

선택 후 절차·마커 기록은 [SKILL.md](SKILL.md), 워크플로별 명령·전환은 [REFERENCE.md](REFERENCE.md).
