# perso-organic — 실측 교훈 (SKILL.md §참조)

> 1~4회차 누적 실측. 검증 설계·디버깅 때 Read.

## 1회차 — dev-workflow 온보딩 문서 (2026-06-05)

- **소비자 경험을 소유하는 페르소나를 제작 라인에 넣어라.** 충실도·교열 QA만으로 짜면 "정확하지만 보기 싫은" 산출물이 나온다. 시각 산출물이면 UX·PMM 류를 검증이 아니라 **제작 단계**에 배치.
- **시각 산출물의 QA 입력은 스크린샷이다.** 코드만 읽는 QA는 첫인상·과밀·어포던스를 못 잡는다. 감독이 렌더 스크린샷(첫 화면·인터랙션 상태·모바일)을 찍어 작업 폴더에 두고 QA가 Read로 보게 하라.
- **감독이 실측한 사실은 기각 불가로 주입.** 패널 위치·클릭 동작 같은 측정 결과를 QA/편집장 프롬프트에 "감독 실측 — 기각 불가"로 명시하면 판정이 흔들리지 않는다.
- **교열 페르소나에는 현실 레퍼런스 테스트를 줘라.** "번역체를 잡아라"로는 부족 — "이 문장이 실제 한국 기술서·공식 가이드에 그대로 실리는가?"라는 판정 질문 하나가 AI 말투·직역 명사까지 잡는다.

## 2회차 — oneul-nutri 마일스톤 PRD (2026-06-06)

- **부서부터 떠올려라 — 역할부터가 아니라.** 제네릭 3역할(리서처/적대리뷰어/편집장)을 6단계 내내 재탕했더니 방 안에 PMM·그로스PM·BD·CPO·도메인 SME가 아무도 없었다. **빠진 직무 = 빠진 비판.** 단계별 오너 부서를 먼저 정하고 그 부서의 실직무를 배치하라.
- **게이트는 엄밀성을 검증하지 적절성을 검증하지 않는다.** 적대QA가 인용·라벨만 보면 *정직하게 라벨링된 전략적 오판*을 통과시킨다(핵심 베팅을 가장 약한 [추정] 증거에 걸어도 못 막음). 적대QA·메인검증에 **전략 렌즈** 의무: "이 결론이 [사실] 다리 위인가 [추정] 다리 위인가 / 이걸 반대할 직무가 편성에 있었나."
- **근거 무거운 단계엔 인용 웹 재검증을 못박아라.** "리뷰어가 직접 WebSearch로 출처·저자·연도·주장일치 재확인"을 의무화하니 매 단계 실제 오류(저자명 곡해·출처↔주장 모순·측정시점 오귀속)를 잡았다. 비용 대비 값짐.
- **게이트형 *직렬* 파이프라인도 1급 패턴이다.** 단계가 직렬·Head 게이팅이면 병렬 Workflow를 고집 말고 "실무→적대QA(블로커 루프 max2)→편집장→Head 게이트" 직렬 루프로 돌려도 된다(호스트 강등 순차 수행과 동형).
- **happy-path 데모 금지.** 시각 프로토타입 QA는 *실패/엣지 케이스*(오인식·경고상태·연속실패)를 데모에 **요구**해야 한다. 성공 장면만 그린 데모는 "구린지조차 판단 불가"하게 만든다.

## 3회차 — perso-organic 스킬 자기정련 (2026-06-06)

- **서브에이전트 vantage 한계를 총책임이 보정하라.** 오케스트레이션 SME(middle)가 "이 환경엔 Workflow 툴이 없다"고 [사실] 단정했으나, 그건 *서브에이전트엔 Workflow 가 안 보이기* 때문이었다(메인 루프는 실제로 돌림). middle 실무가 잡을 수 있는 실수를 했고 high 총책임이 잡았다 — 모델 티어링(실수 전파 비용) 명제의 실증.
- **substrate 중립 + 강등 경로를 1급으로.** "Workflow=특정 제품 전용" 단정 대신 "오케스트레이터 있으면 그것 / 없으면 Task 서브에이전트로 강등"으로 일반화해야 타 호스트(Codex·Gemini)에도 portable.

## 4회차 — oneul-nutri 트렌드/매체 조사 (2026-06-07)

- **리더를 함부로 겸임하지 마라 — 분해형 작업은 커버리지 맵부터.** 총책임이 리더(리서치 팀장)를 "단순하다" 판단해 겸임하고, 맵 없이 거시 토픽으로 실무를 즉흥 분할했더니 **'20대 실유행 앱·코어별 각도'가 통째 누락**(사용자가 잡음). 리더 페르소나를 신설해 *dispatch 전* 3코어×트렌드층 격자를 그리게 하니 빈칸이 즉시 드러났다. **리서치·감사·다각도 = 분해형 작업은 리더가 커버리지 맵부터, 빈칸만 실무로 채운다. 즉흥 dispatch = 반응적 땜빵.**
- **팩트체커의 *정정*도 환각일 수 있다.** 웹사실검증 QA가 미검증 수치(MFP "18% 오차")를 잡고는, 정정 출처로 `ai-food-tracker.com`이라는 **그럴듯한 가짜 사이트를 지어내** "확인됨"으로 통과시켰다. 1차본만 의심하지 말고 **정정값의 출처 실재까지 총책임이 본다** — 안 잡히면 [추정]/[모름]으로 강등(가짜 출처 인용 금지). (1·2회차 '인용 웹 재검증'의 연장: 검증의 산출물도 검증 대상.)
- **blocker 루프의 '재검'을 생략하지 마라.** 정정(FX) 후 grep으로 글자 바뀐 것만 보고 통과시키려다 사용자가 "한 번 돌고 끝 아니지?"로 잡음. 정정은 *재검증*까지가 한 루프(max2) — 정정 중 새 환각이 흔하므로 재검이 본체다.
