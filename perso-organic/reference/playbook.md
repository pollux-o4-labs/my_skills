## 진입 — SERVICE_CONTEXT 수집 디테일

리더를 부르기 전에 도메인 근거 자료를 모아 `SERVICE_CONTEXT` 블록을 만든다: *대상 · 핵심 컨셉/제약 · 작업 목적*.
- **있으면 Read**: 사양·기존 조사·교범·페르소나 파일 등. **없으면** 목적 확인 답변으로 구성.
- **[MUST] 파일로 저장.** SERVICE_CONTEXT 를 `.perso-work/<범위>/service-context.md` 에 쓰고 에이전트엔 *경로만* 줘 Read. **SME 명단·조직도 스케치는 넣지 않는다**(리더 앵커링). (폴더 규칙·전 산출물 동일 — §3.)
- **[근거]** 이게 없으면 리더가 "왜 이 일을 하는지" 모른 채 범용 분해를 한다. 예: 비행기 수리에서 SERVICE_CONTEXT(기종·고장 증상·정비교범·규제)가 없으면 "비행기 전문가 1명" 호출(박제)이 나온다. 있으면 리더가 부서→도메인 SME(파워플랜트·항전·기체구조·NDT)로 제대로 분해한다.
- 누가 받는지는 §3 주입 매트릭스.

## §1 부서 선정 — 부서 매핑 예시

"이건 어떤 종류의 일인가 → 현실 회사에서 어느 부서가 책임지나"를 먼저 답한다. 역할("리서처")이 아니라 부서/직능("사업기획")부터.

**[예시] 부서 매핑 (← 채우는 슬롯, 고정 명단 아님)**
- 시장·경쟁 조사 → 사업기획 / BD + 데이터분석
- 화면·기능 설계 → 서비스기획 + PO + UX
- 카피·브랜드·메시지 → 마케팅 / PMM + 카피라이터
- 영양·의료·법률 등 전문 도메인 → 해당 SME (영양사·임상·변호사)

**[예시] Phase별 부서 유도** (← 고정 명단 아님): 아이디어=UX리서처·SME / 시장=BD·데이터분석 / 기능=서비스기획·PO / MVP=그로스PM·기술PM / 최종=CPO.

## §2 인력 배치 — 4계층 디테일

| 계층 | 누가 | 하는 일 (디테일) |
|---|---|---|
| **총책임** | main agent (감독) | 부서 선정·분담 승인·최종 검증·유저 소통. 페르소나 연기 금지. |
| **리더** | 오너 부서의 리드 | 분담표·구조 스펙 확정. **별도 워크플로(총책임 겸임 안 함).** 실무 dispatch *전* 커버리지 맵(빈칸 격자)부터 — 생략하면 빈칸을 반응적으로 땜빵. 착수 후 SME 간 결론 충돌 시 리더가 재정 or 총책임 에스컬레이션 — 통합 단계로 미루지 않음. Phase 오너 부서 바뀌면 리더도 재고용. |
| **실무** | 도메인 SME 페르소나들 | 실제 산출물 제작. 매 작업 새로 고용되는 핵심 계층 — 도메인 직무를 그때 채운다. [예시] PMM·그로스PM·BD·데이터·UX·영양사 …(박제 명단 아님). |
| **리뷰** | 적대형 QA + 통합(편집장) | 적대QA = 흠을 찾아 반박하는 *공격자*. 통합 = 종합·수정지시·판정·환원 리포트. 공격자 ≠ 판정자 2직능 분리. |

### 리더 협상 루프 (필수 패턴 — 모든 perso-organic 작업)

리더가 파일만 저장하고 끝나면 총책임과 연결이 끊긴다. execute 스크립트가 리더 결정을 무시한 채 하드코딩 값으로 돌아가는 **구조적 단절** — 리더가 7번째 경쟁사를 추가해도 execute는 6개로만 돈다.

**[MUST] 유기적이되 앵커링 금지.** 목적은 *총책임 초안 ↔ 리더 최상안*을 합쳐 더 좋은 조직도를 얻는 것. 총책임이 조직도 초안을 들 수는 있으나(선택), **리더 프롬프트엔 SERVICE_CONTEXT만 주고 그 초안은 주입하지 않는다** — 주면 리더가 자기 최상안 대신 내 안을 보정하는 데 그쳐 fresh한 분해를 잃는다. [근거] 직접 실수: 총책임이 SME 4명 초안을 *확정안처럼* plan 게이트에 펼치고 dispatch하려 함 → 리더 협상·앵커링 회피 둘 다 위반.

(단계 1~4 = SKILL §진입 ①②③④. 0은 선택 사전단계.)

0. **(선택) 총책임 초안** — 검토용으로만 보유. 리더·plan 게이트에 *확정안처럼* 펼치지 않는다(plan은 "리더 dispatch" 승인까지만).
1. **리더 워크플로: 백지에서 `staffingProposal` 도출**해 총책임에 반환(파일 저장과 별개). schema 필드 = `reference/workflow-skeleton.md` `STAFFING` (scopeItems · smeConfig · qaConfig · riskFlags). *이 시점엔 조직도(인력)까지 — 커버리지 맵은 아직.* — 반환 없거나 필수필드(scopeItems·smeConfig) 누락 시 리더 1회 재호출, 재실패면 사용자 보고·중단.
2. **총책임이 반환값 ↔ 초안 대조·보완·합의**: Q1~Q4 반박제 재확인(제네릭 SME 반려) / 누락 scope·표면 보완 / 경험칙상 필수 검증자(팩트체커·논리QA·적대QA)는 리더가 빠뜨려도 총책임이 넣음 / Q4 미충족 시 반대 입장 QA 추가. → **조직도 확정([MUST] 사용자 재승인 — 없으면 3·4 진행 금지).**
3. **확정 인력으로 커버리지 맵 작성**(리더 태스크) — 각 SME가 채울 칸 격자. *빈칸 발견 = 인력 추가하러 2로 되돌아감*(종료: 빈칸 0 또는 총책임이 "미커버 허용" 판단 시 맵 확정). 맵 완성 후 SME dispatch.
4. **합의된 `staffingConfig`를 execute `args`로 전달 → `args.smeConfig` 배열로 동적 생성**(하드코딩 금지).

### 핵심 규칙 근거

- **실무 계층 비우지 마라:** [근거] 2회차 실측 — PMM·그로스PM·BD·CPO·도메인 SME 부재 → 전략·잔존·시장 비판 통째 누락. → `lessons.md` 2회차.
- **검증자 3종 / "최대 2" 상한은 ③에만:** [근거] ③ 4명 동시 → 라운드당 finding 24~28개로 판정 루프만 늘어지고 적중은 2렌즈 집중. ①②(팩트체커·논리QA)를 ③ 예산에 합치면 "영원히 팩트체커+적대1"로 굳어 도메인 렌즈가 못 들어온다. → `lessons.md`.
- **SME 수·통합자 조건부:** SME 수 = `smeConfig` 길이(1~N), "2명"은 예시. 통합자는 *병합할 raw 있을 때만*(SME 2명↑). 1명이면 빼고 판정=리더·환원=총책임 흡수. 적대QA는 SME 수 무관 항상 유지.
- **소비자 경험 오너는 실무에:** [근거] UX·PMM 류를 QA 로만 쓰면 "정확하지만 보기 싫은" 결과. → `lessons.md` 1회차.

### 모델 티어링 (실수 전파 비용 기준 — 3티어 이산)

각 페르소나는 *자기 계층의 실수 전파 비용*으로 티어를 고른다. 아래는 휴리스틱 기본값 — 라벨만 보고 자동 배정 금지(그 자체가 박제).

- **low** — 단순 추출·정리·형식 변환. 실수 전파 비용 최소.
- **middle** — 실무 기본값. *실수는 리뷰가 잡는다* 전제가 서는 작업.
- **high** — 실수가 최종물 직행하는 판단 계층: 리뷰(적대QA·편집장)·총책임.
- **예외 상향** — 실무 산출이 품질 천장을 결정하고 리뷰가 못 메우는 작업(창작·카피·디자인 방향 등)은 실무도 high.
- **이산만** — 같은 모델 내 effort 하위분할 없음. 3티어 중 하나.
- 호스트 매핑: Claude=haiku(low)/sonnet(middle)/opus(high). 타 호스트는 동급 모델로 자율 매핑.

## §3 실행 — 검증 흐름 디테일

**정석 흐름**: `실무 → 팩트체커(영역별, sonnet) → 원 SME 재가동 → 통합 → 적대QA → 고치기 → 팩트체커가 고친 부분만 재검`.

- **팩트체커(ctx✗, 기본 sonnet):** 수치·출처·내부논리만 — SERVICE_CONTEXT 주지 않음(맥락 주면 편향). **모델은 sonnet 기본**(단순 대조라 충분), 규제·안전처럼 틀리면 큰일 나는 항목만 opus. high 발견 시 **원 SME 재가동**(통합자가 직접 안 고침).
- **팩트체크는 두 번:** ①실무 직후(틀린 수치를 합치기 전에 차단) ②고치고 난 직후(고치는 과정에서 새 오류·지어낸 출처가 끼는 게 실측 확인됨). 고치기 → 다시 검증까지가 한 묶음이고, **재검증은 팩트체커**(편집장 아님)가 고친 부분을 다시 보는 것 — 편집장이 글자 바뀐 것만 보고 넘기면 구멍. → `lessons.md` 4회차.
- **SME 재가동(ctx✓):** 지적 부분을 *원 주장 맥락 유지하며* 고침. raw + 통합본 *둘 다* 수정(통합자 재실행 X). [근거] ctx 없이 고치면 "수치만 갈아끼우고 A↔B 관계 상실".
- **통합(편집장):** QA 전체 barrier 종합 → high 있으면 수정 라운드 → 해당 QA만 재검(max 2). 남으면 유저 보고 후 멈춤. SME 1명이면 생략.
- **프롬프트 4요소 + 주입 매트릭스:** 역할 정체성 · 담당 영역 · 산출물 형식 · 참조 입력 경로. SERVICE_CONTEXT 선택 주입 — 리더·실무·통합·갭분석(리더 태스크, 별도 계층 아님) = ✓ / 팩트체커·논리QA = ✗.
- **실행 수단:** 호스트가 Workflow 류 오케스트레이터 제공하면 그것으로, 없으면 `Task` 서브에이전트로 강등(blockedBy 그래프, 작업폴더 파일). 강등 시 저널 캐시·resume·병렬 포기. 노출 여부 먼저 확인.
- **게이트형 직렬도 1급:** 단계가 직렬·Head 게이팅이면 "실무→적대QA(블로커 max2)→편집장→Head 게이트" 직렬 루프 OK. 페이즈 구성은 STAGES config 가변부.
- **[MUST] 모든 단계 산출물은 `<산출물 폴더>/.perso-work/<범위>/` 로** (작업별 하위 신설 — 같은 산출물 폴더의 다른 작업과 섞임 방지). SERVICE_CONTEXT·커버리지 맵·각 SME raw·통합본·QA finding 전부 파일로. 에이전트는 *경로 받아 Read*, 반환값엔 요약만.
- **resume:** 수정·중단 후 처음부터 재발사 금지 — 안 바뀐 호출은 캐시 반환(캐시 키=프롬프트 문자열 동일성, 같은 세션 한정). 과신 금지.
- 의사코드·schema·강등 substrate 상세 → `reference/workflow-skeleton.md`.

## §4 검증 — findings 라우팅 + 유저 2모드

**findings 라우팅** — 총책임이 severity·type 으로 재실행 경로를 가른다. QA·통합은 finding 에 `severity(blocker|advisory)` + `type(content|direction)` 을 단다.
- `blocker` = 반드시 처리 / `advisory` = 재량 skip.
- `type=content`(수치·내용 오류) → SME 재가동(워크플로 내, raw+통합본).
- `type=direction`(프레임·scope 자체가 틀림) → 실무 워크플로 재트리거(편성/커버리지 재검토 후). 워크플로 재실행은 총책임만.
- QA 자체엔 무한 블로커 루프 없음 — 루프·재실행은 총책임 판단(편집장 수정 라운드 max 2 별개).

**유저 검증 2모드** (혼합 산출물은 영역별 분리):

| 모드 | 대상 | 방법 |
|---|---|---|
| **논리형** | 기술문서·번역·데이터 | 중학생 단어로 분해 → "A=B, C=D, 따라서 b→c" 논리 사슬이 말 되면 OK. 도메인 지식 아닌 *논리*로. |
| **주관·시각형** | UX·디자인·카피 | 적대형 QA 거친 뒤 → 유저가 소비자 입장 첫인상으로 OK. |
