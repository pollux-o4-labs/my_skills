# perso-organic — 설계 컨텍스트 (grill 완료 · v1 구현됨 2026-06-05)

> 모르는 도메인 작업을, AI가 동적 편성한 "전문 페르소나 조직"에 병렬 위임하는 스킬.
> 사용자는 감독·통합·빌드·검증. main agent는 감독 + 사용자와 함께 검증.

## 용어 (glossary)

- **페르소나(persona)**: 특정 전문 역할을 입은 subagent 1개. (예: 카피라이터, 번역가, 정보설계자)
- **페르소나 조직(persona org)**: 한 작업을 위해 편성된 페르소나들의 집합 + 그들의 영역 분담 구조.
- **편성(staffing)**: AI가 도메인을 보고 어떤 페르소나가 몇 명, 어떤 영역으로 필요한지 정하는 행위. 이 직무를 맡는 계층이 **리더**(구 명칭 "Strategy" — 폐기, SKILL.md §2 통일 용어).
- **영역(domain boundary)**: 페르소나 간 충돌 방지를 위한 작업 분담 경계.
- **감독(main agent)**: 페르소나를 호출하지 않고, 편성·검증·사용자 소통을 맡는 상위 에이전트.

## 확정된 결정

- **D1. 명단은 동적 편성.** 고정 명단 금지 — 모르는 도메인은 필요 역할도 미리 모름.
  단, QA·통합 같은 골격 역할은 항상 포함(품질 하한선).
- **흐름(초안):**
  1. 문제 발생 → 사용자가 도메인 모름(마케팅/글쓰기/번역 등)
  2. AI가 짧게 grill-me식으로 "진짜 원하는 것/목적" 확인
  3. AI ↔ 사용자 방향(목적) 일치
  4. AI가 적절한 페르소나 조직 편성
  5. main agent = 감독 + 사용자와 검증
  6. 페르소나 조직이 병렬로 빠르게 처리

- **D2. 신뢰는 "사용자 검증 가능 형태로 환원"으로 확보.** 도메인별 2모드:
  - **논리형(기술문서 등)**: 중학생 수준 단어로 세부 분해 → A=B, C=D, b→c 식 논리 사슬이 말 되면 사용자 OK. (도메인 지식 아닌 *논리*로 검증)
  - **주관/시각형(UX 등)**: 전문 페르소나 검증 거친 뒤 → 사용자가 소비자 입장 첫인상으로 OK.
  - 따라서 산출물은 항상 "왜 이렇게 했나 + 근거 + 불확실"을 환원해 첨부.
- **관찰**: 이전 실제 흐름은 Workflow(리더→실무 병렬→리뷰→통합)로 구현됨, 231k 토큰.
- **D3. 구현은 Workflow 기반.** 스킬 = "도메인 보고 페르소나 조직 짜서 Workflow 스크립트로 emit"하는 메타 역할. (즉석 subagent 호출은 채팅으로 이미 가능 → 스킬 가치는 조직형 병렬 파이프라인에 있음)

- **D4. 영역 분담은 자동.** 리더가 분담표 확정 → 각 페르소나는 자기 영역 파일/섹션만 수정(병렬 충돌 방지, 산출물 단위로 미리 분리). **분담 자체엔 사용자 승인 게이트 없음**(메인+편집장 2중인데 사용자까지 끼면 병목). ※ D5의 *편성 plan 게이트*(편성안 전체를 plan mode 로 승인)와 별개 — 그건 착수 전 1회, D4 는 분담 단계 무게이트.
- **D5. 목적-일치 게이트 = plan mode.** 별도 grill 단계 아님. 메인이 "목적+편성안"을 plan으로 작성 → plan mode 승인 → Workflow emit. (스크린샷 예시가 바로 돈 건 사전 유사작업 + plan mode 수락 때문)
- **D6. grill 의존성 인라인.** 풀 `grill-me` 스킬 의존 금지(미설치 환경서 깨짐). "요청이 plan 짜기도 모호하면 핵심 2~3개만 묻고 멈춤" 규칙만 SKILL.md에 직접 박음.

- **D7. 내부 QA = 적대형.** 흠을 찾아 반박하는 공격자로 프롬프트("목적 달성 못 하는 이유를 찾아라"). 그럴듯한-틀림 방지. 수정 라운드 max 2 — 그 뒤에도 high-severity 남으면 유저 보고 후 정지. (2026-06-05 확정) **+2026-06-10: 검증자를 3종으로 분화** — ①팩트체커 ②논리QA(둘 다 ctx✗ 고정) ③도메인 적대 렌즈(협상). "동시 최대 2" 상한은 ③에만. *규칙 본문은 SKILL.md §2 — 여기선 결정 사실만.*
- **D8. 골격 역할 = 리더(편성) + 적대형 QA + 통합(편집장).** 리더·적대QA는 항상 포함, 나머지는 동적 편성. **통합자는 조건부**(SME 1명이면 생략) — 결정 사실만, *규칙 본문은 SKILL.md §2.* (2026-06-05 확정 / 통합자 조건부 2026-06-10 보완) **+2026-06-11: "단순 작업이면 main 겸임" 폐기 — 비분해/분해 구분 제거(스킬은 manual-only라 호출 시 항상 조직형). 리더는 *항상 별도 워크플로*(유기적 편성 = 워크플로 2개).**

- **D9. 팩트체커 = 기본 sonnet + 두 번 검증.** 모델은 sonnet 기본(단순 대조라 충분), 규제·안전처럼 틀리면 큰일 나는 항목만 opus(습관적 opus 배정 금지). 팩트체크는 ①실무 직후 ②고치고 난 직후 두 번 — 고친 부분은 팩트체커가 따로 다시 검증(편집장이 글자만 훑고 통과 금지). 결정 사실만, *규칙 본문 SKILL.md §3.* (2026-06-10 eCRF 테스트 실측)
