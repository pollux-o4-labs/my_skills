# perso-organic — 동적 페르소나 조직

모르는 도메인 작업을, **그 작업에 맞는 실제 조직을 그때그때 고용**해서 병렬 위임한다.
main agent는 페르소나를 직접 연기하지 않는다 — **총책임(감독)**으로서 부서 편성·검증·유저 소통만 맡는다.

> **읽는 법.** 이 파일 = *척추*(규칙·표·안티패턴). 각 단계의 *실행 디테일·근거·예시*는 `reference/playbook.md` 에 분리 — **단계 진입 시 해당 절만 Read**.
> `[MUST]`=불변 규칙(어기면 안티패턴) / `[예시]`=가변 슬롯(박제 명단 아님, 디테일은 playbook).
> **자주 쓰는 말.** SME=도메인 실무 전문가 · SERVICE_CONTEXT=작업의 대상·목적·제약을 담은 맥락 묶음 · ctx✓/✗=그 맥락을 주입함/안 함 · raw=SME 가 처음 만든 개별 파일 · execute=3단계 실행(워크플로) · staffingProposal=리더가 총책임에 돌려주는 편성안(필드는 `workflow-skeleton.md` STAFFING).
>
> **[MUST] 골격은 *기능*이지 *명단*이 아니다.** 편성·검증·통합은 매번 있는 *기능*(리더·리뷰·총책임), 실무는 도메인에서 유도되는 *부서*다. "편성1·QA1·통합1"을 직무명으로 박제하면 방 안에 진짜 전문가가 0명이 된다.

## 진입 게이트 (4단계 실행 전, 최상단)

> **[MUST] 1·2단계(부서·인력)는 리더 워크플로로 돈다 — plan 게이트는 그 *전*.** 즉 plan 승인 대상 = 확정 SME 명단이 아니라 **'리더 워크플로 dispatch'**(SME 확정은 아래 ②). 3단계(실행)는 ② 재승인 후.
> **[MUST] 편성은 *유기적*(워크플로 2개).** 총책임 초안은 *선택*이고 **리더에 먼저 주입 금지**(앵커링 — 리더가 자기 최상안 대신 내 안 보정에 그침). ① 리더 워크플로가 SERVICE_CONTEXT만 받아 *백지에서* staffingProposal 도출 → ② 총책임이 초안·경험칙(필수 검증자 등)과 대조·합의 후 **조직도 확정(사용자 재승인 [MUST] — 없으면 ③·execute 금지)** → ③ 확정 인력으로 커버리지 맵 → ④ execute. (SKILL ①②③④ = playbook §2 단계 1·2·3·4.) 디테일 → playbook §2.

1. **목적 확인.** 모호하면 **핵심 질문 2~3개만 묻고 멈춘다**(외부 grill 의존 금지). 확인: 원하는 결과물 / 누가 소비 / 성공 기준.
2. **[MUST] SERVICE_CONTEXT 수집 — 리더 부르기 전.** 도메인 근거(사양·교범·기존 조사·페르소나 등 Read, 없으면 목적 답변)를 *대상·핵심 컨셉/제약·작업 목적* 블록으로. 없으면 리더가 범용 분해(→ "비행기 전문가 1명" 박제). **디테일·예시 → `playbook.md` §진입.** 누가 받는지는 §3 주입 매트릭스.
3. **plan 게이트.** plan mode 로 "목적 + 편성안" 쓰고 ExitPlanMode 승인. 필수: 목적 1줄+성공 기준 / **편성표** / 산출물 경로·형태 + 검증 모드. **편성표 = 부서·목적·잠정 표면·티어 범위까지만**(SME 명단은 리더 staffingProposal 후 ② 재승인에서 확정 — 위 유기적 편성).
4. **[MUST] 반박제 4문 — 하나라도 No 면 편성표 반려.**
   - **Q1** 실무 명칭이 *도메인 무관 제네릭*인가? ("UX리서처"=통과 / "리서처·리뷰어·에디터"=불합격)
   - **Q2** 실무 명칭이 1단계 부서/직능에서 유도되나? (추적 불가면 박제)
   - **Q3** 직전 Phase 실무 명단과 다른가? (부서 바뀌었는데 실무 같으면 박제. 첫 Phase 는 Q1·Q2·Q4만)
   - **Q4** 이 결론을 *반대·책임질* 직무가 실무 또는 QA 에 있나? (전략 공백 — 4단계 전략 렌즈와 단일 질문, 지금 묻는다)
   - **[근거] 식별 신호:** 실무 칸에 도메인 명사 없고 제네릭만 보이면 박제.

## 4단계 루프 (작업/Phase 마다 1회전)

| # | 단계 | 한 줄 | exit 조건 | 디테일 |
|---|---|---|---|---|
| 1 | **부서 선정** | 이 일의 오너가 현실에서 어느 *부서/직능*인가. 역할 아니라 부서. | 오너 부서 1~N개 확정. | playbook §1 |
| 2 | **인력 배치** | 부서의 실직무를 **총책임–리더–실무–리뷰** 4계층으로 고용·분담 + 티어. | staffingConfig 확정(리더 staffingProposal → ② 사용자 재승인). 반박제 4문 통과. | playbook §2 |
| 3 | **실행** | 편성표를 오케스트레이터로 돌린다(직렬/병렬 + 적대QA 블로커 루프). | high-severity 0 또는 max 2라운드. | playbook §3 |
| 4 | **메인 검증** | 총책임이 *엄밀성 + 전략 렌즈*로 최종 검증 + 환원 리포트. | 환원 리포트 + 2모드 검증 완료. | playbook §4 |

> ⚠️ **[MUST] Phase 마다 다시 돈다.** 프로젝트 전체에 조직 하나 박제 금지 — Phase 가 바뀌면 오너 부서가 바뀐다.

## 2단계 — 인력 배치: 4계층 + 핵심 규칙

> 4계층 cell 디테일·**리더 협상 루프**·부서 매핑 예시·모델 티어 설명 → **`playbook.md` §1·§2.**

| 계층 | 누가 | 핵심 [MUST] |
|---|---|---|
| **총책임** | main agent (감독) | 편성·검증·유저 소통. **페르소나 연기 금지.** |
| **리더** | 오너 부서 리드 | 분담표 확정. **[MUST] 백지에서 staffingProposal 도출 — 총책임 초안 주입받지 않음(앵커링 방지).** **커버리지 맵은 조직도 합의·확정 후** SME dispatch 전에 작성(빈칸 = 인력 추가 협상). **[MUST] 파일 저장만 하고 끝내지 말고 `staffingProposal` schema 를 총책임에 반환** — 안 하면 execute 가 리더 결정 무시·하드코딩으로 돈다. |
| **실무** | **도메인 SME 들** | 산출물 제작. **[MUST] 매 작업 새로 고용되는 핵심 계층 — 비우지 마라.** |
| **리뷰** | 적대QA + 통합(편집장) | **공격자 ≠ 판정자 2직능 분리.** **[MUST] 통합은 high-severity 단독 기각 불가 — 총책임 승인 에스컬레이션.** |

**핵심 규칙 (근거·예시 → `playbook.md` §2)**

- **[MUST] 실무 계층 비우지 마라.** 제네릭 "리서처/리뷰어/편집장" 재탕 = 방 안에 진짜 전문가 0명. 리더/리뷰는 *기능*, 실무는 *부서*(매번 새로).
- **[MUST] 검증자 3종 — "최대 2" 상한은 ③에만.** ①팩트체커(ctx✗, 수치·출처) ②논리QA(ctx✗, baseline) = 고정 검증자, 상한 예산 밖. ③도메인 적대 렌즈(리더 협상) = **동시 최대 2**, 초과분 다음 라운드. 리더 제안 도메인 QA 는 ③에서 추가(관점 다르면)·병합(겹치면).
- **[MUST] SME 수·통합자는 상수 아님.** SME 수 = `smeConfig` 길이(1~N), "2명"은 예시. **통합자는 조건부** — SME 2명↑(병합할 raw 있을 때)만 신설, 1명이면 빼고 판정=리더·환원=총책임 흡수(적대QA 는 항상 유지).
- **[MUST] 분담은 자동.** 리더가 분담표 확정 → 각 페르소나는 자기 영역만. 분담(*SME 간 영역 쪼개기*)에 유저 승인 게이트 끼우지 마라(병목). *단 ②의 조직도 확정(누가 방에 있나)은 별개 — 사용자 재승인 대상.*
- **[MUST] 소비자 경험 오너는 검증 아니라 *실무(제작)* 에.** QA 로만 쓰면 "정확하지만 보기 싫은" 결과.
- **[MUST] 모델 티어링 (실수 전파 비용 기준, 3티어 이산).** low=추출·정리 / middle=실무 기본(실수는 리뷰가 잡음) / high=판단 계층(리뷰·총책임, 실수 최종물 직행). **예외 상향**: 창작·카피·디자인 등 실무가 품질 천장이면 실무도 high. **라벨만 보고 티어 자동배정 금지.** (Claude=haiku/sonnet/opus. 설명 → playbook §2.)

## 3단계 — 실행: [MUST] 원칙 (메커니즘은 playbook)

> 실행 수단(Workflow/Task)·프롬프트 4요소·작업폴더·resume·게이트형 직렬·통합 max2 등 *메커니즘*과 검증 흐름·매트릭스 디테일 → **`playbook.md` §3** / 의사코드·schema → **`workflow-skeleton.md`.** 아래 본문엔 *어떤 경우에도 남는 원칙*만.

- **[MUST] 조직형 파이프라인으로 실행.** 즉석 subagent 나열이 아니라 오케스트레이터로 돌리는 게 이 스킬의 가치 (Workflow 류, 없으면 `Task` 강등 — 둘 다 1급).
- **[MUST] 팩트체커(ctx✗, 기본 sonnet) — 2지점·고친 부분만 재검.** 수치·출처만, SERVICE_CONTEXT 주지 마라. 모델 sonnet 기본(규제·안전 항목만 opus). high 발견 시 통합자 아니라 *원 SME* 가 raw+통합본 수정. 팩트체크는 ①실무 직후 ②고친 직후 두 번 — 재검증은 팩트체커(편집장 아님)가 고친 부분 다시 보기(글자만 훑고 통과 금지).
- **[MUST] SERVICE_CONTEXT 선택 주입 — 인라인 아닌 *파일 경로*로.** SERVICE_CONTEXT·중간 산출물은 `<산출물 폴더>/.perso-work/<범위>/` 에 파일로, 에이전트엔 경로만 줘 Read. 주입 대상: 리더·실무·통합·갭분석(리더 태스크, 별도 계층 아님)=✓ / 팩트체커·논리QA=✗. 전 계층 동일 주입은 안티패턴.
- **[MUST] execute는 협상 결과를 받는다(하드코딩 금지).** scope·SME·QA 를 스크립트에 박으면 리더 협상 무시 — `args.smeConfig` 배열로 config-driven 생성.

## 4단계 — 메인 검증 (엄밀성 + 전략 렌즈)

> findings 라우팅·유저 2모드 표 → **`playbook.md` §4.**

- **[MUST] 모든 산출(리뷰·편성안·코드 포함)에 환원 리포트.** 왜 이렇게 했나 + 근거 + 불확실 지점.
- **[MUST] 엄밀성:** 인용·라벨·논리 무결성. 단 *엄밀성만 보지 마라* — 게이트는 적절성을 안 본다.
- **[MUST] 전략 렌즈:** "핵심 베팅이 가장 약한 [추정] 다리 위인가 / 이 결론을 반대·책임질 직무가 편성에 빠졌나." 정직하게 라벨링된 *전략적 오판*은 엄밀성 검사를 통과한다 — 이걸 막는 게 전략 렌즈(진입 Q4 와 단일 질문).
- **[MUST] findings 라우팅:** finding 에 `severity(blocker|advisory)`+`type(content|direction)`. content→SME 재가동 / direction→실무 워크플로 재트리거(**총책임만**). 디테일 → playbook §4.

## 안티패턴 (전부 위 [MUST] 위반 — 빠른 역(逆)체크)

페르소나 직접연기 · 명단 박제(실무 SME 0명, 제네릭만) · 실무 계층 공석 · 리더 박제(Phase 바뀌었는데 재사용) · 셀프-기각(통합이 high 단독 합격) · **통과형 QA**(체크리스트 만족=합격, 그럴듯한-틀림 못 잡음) · 모델 티어 무시/고정 박제 · 분담표 없이 병렬 발사 · execute 하드코딩(리더 협상 우회) · **리더 앵커링(총책임 초안을 리더에 선주입)** · **총책임이 SME 명단을 리더 없이 확정** · 환원 리포트 생략.
