# perso-organic — 동적 페르소나 조직

모르는 도메인 작업을, AI가 그때그때 편성한 "전문 페르소나 조직"에 병렬 위임한다.
main agent는 페르소나를 직접 연기하지 않는다 — **감독**으로서 편성·검증·유저 소통만 맡는다.

## 용어

- **페르소나**: 특정 전문 역할을 입은 subagent 1개 (예: 카피라이터, 번역가, 정보설계자)
- **페르소나 조직**: 한 작업을 위해 편성된 페르소나 집합 + 영역 분담 구조
- **편성(staffing)**: 도메인을 보고 어떤 페르소나가 몇 명, 어떤 영역으로 필요한지 정하는 행위
- **환원 리포트**: 산출물에 첨부하는 "왜 이렇게 했나 + 근거 + 불확실 지점" 설명

## 플로우 (5단계)

### ① 목적 확인

요청이 plan을 짜기에도 모호하면 **핵심 질문 2~3개만 묻고 멈춘다**. 더 묻지 않는다.
(외부 grill 스킬에 의존하지 말 것 — 미설치 환경에서 깨진다.)

확인할 것: 진짜 원하는 결과물이 뭔가 / 누가 소비하나 / 성공 기준이 뭔가.

### ② plan mode 게이트

**별도 승인 단계를 만들지 않는다.** plan mode로 "목적 + 편성안"을 plan 파일에 쓰고
ExitPlanMode로 승인받는다. plan에 반드시 포함:

- 목적 1줄 + 성공 기준
- **편성표**: 페르소나 이름 · 역할 · 담당 영역 · Phase (표 형식)
- 산출물 경로와 형태
- 검증 모드 (논리형/주관·시각형 — ⑤ 참조)

### ③ 편성 규칙

- **명단은 동적.** 고정 명단 금지 — 도메인을 보고 그때 정한다.
- **골격 3역할은 항상 포함** (품질 하한선):
  1. **편성(Strategy)** 1명 — 분담표·구조 스펙 확정. plan 단계에서 main이 겸하거나, 복잡하면 Phase 1 페르소나로 분리
  2. **적대형 QA** — 통과시키는 검사자가 아니라 **흠을 찾아 반박하는 공격자**. "이 산출물이 목적을 달성하지 못하는 이유를 찾아라"로 프롬프트. 독자 페르소나가 여럿이면 독자별로 1명씩
  3. **통합(편집장)** 1명 — QA 결과 종합 → 수정 지시 → 최종 판정 + 환원 리포트 작성
- **영역 분담은 자동.** 편성이 분담표를 확정하면 각 페르소나는 자기 영역 파일/섹션만 만진다 (병렬 충돌 방지). 분담에 유저 승인 게이트를 끼우지 않는다 — 병목.

### ④ Workflow emit

승인된 편성안을 **Workflow 툴 스크립트로 emit**해서 실행한다. 즉석 subagent 호출 나열이 아니라 조직형 병렬 파이프라인이 이 스킬의 가치다.

> **호스트 분기**: Workflow 툴은 Claude Code 전용. 다른 호스트(Codex·Gemini CLI)에서는 ①~③(목적 확인·plan 게이트·편성)은 동일하게 하되, 실행은 편성표 순서대로 순차 수행으로 강등 — 저널 캐시·resume·병렬은 기대하지 말 것.

기본 형태:

- 제작 페르소나들: `pipeline()` (구조 → 집필 → 구현처럼 의존 있으면 직렬 스테이지, 영역이 갈리면 병렬)
- 적대형 QA: `parallel()` — 전원이 같은 산출물을 동시에 공격, findings + severity를 schema로 반환
- 편집장: QA 전체를 받아 종합 (barrier 정당) → high-severity 있으면 수정 라운드 → 해당 QA만 재검. **max 2라운드** — 그 뒤에도 남으면 유저에게 보고하고 멈춘다
- 각 페르소나 프롬프트에 반드시: 역할 정체성, 담당 영역(만질 수 있는 범위), 산출물 형식, 참조할 입력 파일 경로
- **중간 산출물은 작업 폴더 파일로** (예: `<산출물 폴더>/.perso-work/`) — 반환값엔 요약만. 긴 반환값은 잘릴 수 있고, 파일이어야 다음 페르소나가 읽고 유저도 환원 자료로 확인한다
- **재실행은 resume로.** 스크립트 수정·중단 후 처음부터 재발사 금지 — `Workflow({scriptPath, resumeFromRunId})`로 이어가면 프롬프트가 안 바뀐 페르소나 호출은 저널 캐시에서 즉시 반환되고, 바뀐 지점부터만 실제 재실행된다 (같은 세션 한정)

### ⑤ 검증 — 유저가 확인 가능한 형태로 환원

산출물에는 항상 **환원 리포트**(왜 이렇게 했나 + 근거 + 불확실 지점)를 첨부한다.
유저 검증은 도메인별 2모드:

| 모드 | 대상 | 방법 |
|---|---|---|
| **논리형** | 기술문서·번역·데이터 등 | 중학생 수준 단어로 세부 분해 → "A=B, C=D, 따라서 b→c" 논리 사슬이 말이 되면 OK. 도메인 지식이 아니라 *논리*로 검증 |
| **주관·시각형** | UX·디자인·카피 등 | 전문 페르소나 검증(적대형 QA)을 거친 뒤 → 유저가 소비자 입장의 첫인상으로 OK |

혼합 산출물(예: 기술문서 + 시각 디자인)은 두 모드를 영역별로 나눠 적용한다.

## 안티패턴

- main agent가 페르소나를 직접 연기 — 감독 역할 붕괴
- 모든 작업에 같은 명단 재사용 — D1 위반, 도메인 무시
- 통과형 QA ("체크리스트 만족하니 합격") — 그럴듯한-틀림을 못 잡는다
- 분담표 없이 병렬 발사 — 같은 영역을 두 페르소나가 고쳐 충돌
- 환원 리포트 생략 — 유저가 결과를 믿을 근거가 없어진다
